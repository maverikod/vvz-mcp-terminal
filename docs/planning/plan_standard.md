# Стандарт ТЗ и плана разработки

> Human-readable версия. Будет переведена в машинночитаемый формат после согласования.

---

## 1. Базовая концепция

Разработка ведётся через декомпозицию исходной идеи в иерархию артефактов. На каждом уровне форма представления и объём контекста разные, но сохраняется одно сквозное свойство: **сумма потомков семантически воспроизводит родителя**.

Артефактов пять:

1. **Исходное ТЗ** — human-readable текст с концепциями.
2. **Машинночитаемое ТЗ** — выжимка концепций и связей между ними.
3. **Глобальные шаги (ГШ)** — карта исходного ТЗ + концептуальное описание блоков.
4. **Тактические шаги (ТШ)** — детализация одного ГШ до конкретных объектов и действий.
5. **Атомарные шаги (АШ)** — самодостаточные промпты для модели-кодера.

Уровни 1–2 описывают **что должно быть** в системе. Уровни 3–5 описывают **как это будет реализовано**.

Все изменения проходят строго сверху вниз через каскадную процедуру (см. раздел 8). Прямая правка нижестоящих артефактов в обход вышестоящих запрещена.

---

## 2. Уровень 1 — Исходное ТЗ

### 2.1. Назначение

Один файл свободной формы, в котором набрасываются концепции будущей системы. Это сырьё для всех последующих уровней.

### 2.2. Форма

Markdown, без жёсткой структуры. Допускаются разделы, списки, диаграммы, размышления, альтернативы.

Все строки исходного ТЗ — адресуемые единицы. На них ссылаются нижние уровни через диапазоны строк.

### 2.3. Расположение

```
docs/plans/<plan_name>/source_spec.md
```

### 2.4. Разметка значимых и незначимых строк

Не все строки исходного ТЗ требуют реализации. Преамбула, размышления автора, отвергнутые альтернативы — присутствуют в файле, но не должны попадать в концепции и не должны покрываться ГШ.

Такие фрагменты помечаются inline-разметкой:

```markdown
<!-- non-binding -->
Тут размышления, преамбула, отвергнутые альтернативы.
Эти строки не извлекаются в машинночитаемое ТЗ
и не покрываются ГШ.
<!-- /non-binding -->
```

Все строки **вне** таких блоков считаются значимыми.

### 2.5. Неизменяемость

Технических механизмов защиты нет. Неизменяемость обеспечивается дисциплиной автора и каскадной процедурой правки. История правок ведётся в git.

### 2.6. Свойства

- Файл существует ровно один на план.
- Любая правка значимых строк запускает каскад (см. раздел 8).
- Правка строк внутри `non-binding` блоков каскад не запускает.

## 3. Уровень 2 — Машинночитаемое ТЗ

### 3.1. Назначение

Структурированная выжимка концепций и связей из исходного ТЗ. Выполняет три функции:

1. Компактный контекст для модели-исполнителя на нижних уровнях.
2. Источник истины для терминологии (одно имя — одно понятие).
3. Опорная сетка для проверки инвариантов I1–I3.

### 3.2. Расположение

```
docs/plans/<plan_name>/spec.yaml
```

### 3.3. Содержимое

**Концепции.** Сущности предметной области с поведением или инвариантом. Каждая имеет:

- уникальный идентификатор,
- каноническое имя (без синонимов),
- краткое определение,
- список свойств и инвариантов,
- список диапазонов строк исходного ТЗ, откуда извлечена.

Концепцией является сущность с поведением или инвариантом. Отдельное свойство сущности концепцией не считается, если у него нет собственного поведения (валидация, нормализация, преобразование).

**Связи.** Типизированные отношения между концепциями. Закрытый список типов:

- `uses` — концепция-A использует концепцию-B в своей работе.
- `owns` — концепция-A владеет концепцией-B (lifecycle, ответственность).
- `implements` — концепция-A реализует контракт/интерфейс концепции-B.
- `extends` — концепция-A расширяет концепцию-B (наследование, специализация).
- `depends_on` — концепция-A зависит от существования концепции-B.
- `produces` — концепция-A производит концепцию-B (создаёт экземпляры, генерирует данные).
- `consumes` — концепция-A потребляет концепцию-B (принимает на вход).

Связи вне этого списка запрещены. Если выразить отношение через комбинацию имеющихся типов невозможно — это сигнал, что концепции описаны неточно.

### 3.4. Чего в машинночитаемом ТЗ нет

- Реализационных деталей (файлов, функций, модулей, библиотек).
- Последовательности действий (это уровни ГШ/ТШ/АШ).
- Альтернатив и размышлений (только зафиксированные решения).
- Прозы (только структурированные поля).

### 3.5. Процедура создания

После заморозки исходного ТЗ:

1. **Извлечение концепций.** По всему тексту выделяются сущности с поведением/инвариантом. Каждая получает идентификатор и каноническое имя.
2. **Дедупликация.** Синонимы сводятся к одному имени.
3. **Извлечение связей.** Для каждой концепции фиксируется: с кем связана, как связана (типы из закрытого списка).
4. **Привязка к строкам.** Каждая концепция получает список диапазонов строк исходного ТЗ.
5. **Проверка полноты.** Объединение диапазонов всех концепций должно покрывать все значимые строки исходного ТЗ. Незакрытые строки — либо концепция упущена, либо строки на самом деле незначимы и должны быть помечены `non-binding`.

### 3.6. Эволюция

Прямая правка машинночитаемого ТЗ запрещена. Любое изменение концепций или связей идёт через правку исходного ТЗ и каскадную перепроекцию.

Если в ходе работы над ГШ обнаруживается упущенная концепция — каскад запускается **от исходного ТЗ**: исходник дополняется так, чтобы концепция была в нём явно представлена, затем перепроецируется машинночитаемое ТЗ, затем переоцениваются ГШ.

## 4. Уровень 3 — Глобальные шаги (ГШ)

### 4.1. Назначение

Концептуальные блоки, на которые разбивается план. Каждый ГШ описывает один или несколько связанных объектов и операции над ними.

### 4.2. Расположение

```
docs/plans/<plan_name>/G-NNN-<slug>/README.yaml
```

### 4.3. Концептуальный тест

ГШ описывается через объекты предметной области и операции над ними, **без упоминания конкретных файлов, модулей, функций или классов реализации**. Если в описании ГШ появляются такие упоминания — это просочившаяся тактика, ГШ требует переработки.

### 4.4. Количество ГШ

Обычно не превышает 10. Если получается больше — это **сигнал к ревизии**: либо концептуальный тест применили нестрого, либо проект слишком крупный и должен быть разбит на несколько отдельных планов. Это индикатор, не запрет.

### 4.5. Содержимое ГШ

Обязательные поля:

- `step_id` — идентификатор формата `G-NNN`.
- `name` — краткое имя.
- `description` — концептуальное описание объектов и операций.
- `concepts` — список идентификаторов концепций машинночитаемого ТЗ, которые покрывает этот ГШ.
- `relations` — список связей машинночитаемого ТЗ, реализуемых в этом ГШ.
- `source_ranges` — список диапазонов строк исходного ТЗ, относящихся к этому ГШ. Допускаются несмежные диапазоны.
- `depends_on` — список ГШ, от которых зависит этот ГШ по порядку выполнения. Описывает только порядок работы, не семантические связи.
- `tactical_steps` — список идентификаторов ТШ.
- `status` — текущий статус (см. раздел 9).

### 4.6. Связи между ГШ

Семантические связи между ГШ выражаются **через машинночитаемое ТЗ**, не через внутренние поля самих ГШ. Если ГШ-A и ГШ-B оперируют общей концепцией X, эта общность видна из того, что обе ссылки указывают на одну запись в машинночитаемом ТЗ.

Поле `depends_on` у ГШ описывает **порядок выполнения**, не семантику. Семантика — в машинночитаемом ТЗ.

Это даёт три следствия:

1. Машинночитаемое ТЗ — единая точка истины для всех связей в системе.
2. ГШ остаётся самодостаточным артефактом: чтобы понять его, не нужно читать соседние ГШ.
3. Каскад работает корректно: любое изменение связи между концепциями = изменение машинночитаемого ТЗ = инвалидация всех ГШ.

### 4.7. Инфраструктурные ГШ

Инфраструктурная составляющая (общий логгер, базовая конфигурация, скелет проекта, общие типы и контракты) оформляется как один или несколько отдельных ГШ внутри плана. Эти ГШ обычно создаются и выполняются первыми, поскольку прикладные ГШ опираются на них через `depends_on`.

Инфраструктурные ГШ подчиняются всем общим правилам: концептуальное описание, покрытие диапазонов исходного ТЗ, инварианты I1–I3.

### 4.8. Процедура создания

Двухфазная.

**Фаза А — карта исходного ТЗ.** Исходный файл размечается на блоки: каждый блок — набор диапазонов строк. Один блок может состоять из нескольких несмежных диапазонов.

**Фаза Б — концептуализация.** Для каждого блока пишется ГШ: концепции, связи, операции (всё через ссылки на машинночитаемое ТЗ).

Сумма всех ГШ покрывает все концепции и связи машинночитаемого ТЗ (инвариант I1).

## 5. Уровень 4 — Тактические шаги (ТШ)

### 5.1. Назначение

Детализация одного ГШ до конкретных объектов и действий. Каждый ТШ оперирует конкретными сущностями: классами, модулями, схемами данных, контрактами API. Ещё не файлами и функциями — это уровень АШ.

### 5.2. Расположение

```
docs/plans/<plan_name>/G-NNN-<slug>/T-NNN-<slug>/README.yaml
```

### 5.3. Содержимое ТШ

Обязательные поля:

- `step_id` — идентификатор формата `T-NNN`.
- `parent_global_step` — идентификатор родительского ГШ.
- `name` — краткое имя.
- `description` — описание: какие сущности создаются/изменяются, какие операции реализуются.
- `concepts` — список идентификаторов концепций машинночитаемого ТЗ, на которые ссылается этот ТШ.
- `inputs` — структурированное описание входов (контракты, типы, ожидаемые артефакты).
- `outputs` — структурированное описание выходов.
- `atomic_steps` — список идентификаторов АШ.
- `status` — текущий статус (см. раздел 9).

Точный формат `inputs`/`outputs` фиксируется при переводе стандарта в машинночитаемую форму.

### 5.4. Свойства ТШ

**Покрытие ГШ.** Сумма ТШ полностью реализует родительский ГШ. Каждая концепция и связь, отнесённая к ГШ, реализуется хотя бы в одном из его ТШ.

**Когерентность.** Один ТШ — одна логически связанная единица работы.

**Однозначность ссылок.** Все упомянутые объекты — концепции из машинночитаемого ТЗ, по их идентификаторам, без свободных имён.

**Отсутствие развилок.** В ТШ не остаётся открытых проектных решений. Не должно быть формулировок «можно сделать через X или Y», «по необходимости», «при необходимости». Если развилка возникает — она поднимается на уровень ГШ или исходного ТЗ и разрешается там.

### 5.5. Процедура создания

Для каждого ГШ:

1. Берутся все диапазоны строк исходного ТЗ, относящиеся к этому ГШ.
2. Анализируются повторно — теперь с точки зрения «что конкретно надо сделать».
3. Создаётся набор ТШ, покрывающий ГШ целиком.

## 6. Уровень 5 — Атомарные шаги (АШ)

### 6.1. Назначение

Самодостаточный промпт для модели-кодера. Шаг, который:

1. Логически невозможно разделить.
2. Касается ровно одного файла кода.
3. Содержит весь контекст, необходимый для выполнения.

### 6.2. Расположение

АШ описываются inline внутри `README.yaml` родительского ТШ, под ключом `atomic_steps`.

### 6.3. Принцип атомарности

«Логически невозможно разделить» означает: при разделении получается либо полшага без смысла, либо два шага, один из которых не имеет смысла без другого.

На практике АШ — это единица вида:

- создать файл с определённой структурой,
- создать функцию с определённой сигнатурой,
- добавить метод в класс,
- модифицировать конкретный участок файла по точно описанному правилу.

### 6.4. Правило «один файл — несколько АШ»

Один АШ касается **ровно одного** файла кода. Один файл кода **может** затрагиваться несколькими АШ. Обратное (один АШ → несколько файлов) недопустимо.

### 6.5. Размер файла кода

Файл кода ограничен 350–400 строк. Это не рекомендация, а **проектное ограничение плана**: если по ТШ получается файл большего размера, ТШ требует пересмотра, файл должен быть разбит на несколько.

Стратегия `patch_region` (передача в промпт фрагмента вместо полного файла) **не поддерживается**. Все АШ работают с полным содержимым целевого файла.

### 6.6. Закрытый список операций

- `create_file` — создать новый файл кода.
- `modify_file` — изменить существующий файл кода.
- `delete_file` — удалить файл кода.
- `rename_file` — переименовать или переместить файл кода.

### 6.7. Приоритет внутри одного файла

Когда один файл кода затрагивается несколькими АШ:

- Каждый АШ имеет поле `priority: N` (целое число).
- Внутри одного файла **все приоритеты обязаны различаться**. Одинаковых быть не может.
- АШ выполняются в порядке возрастания приоритета.
- Между файлами приоритеты не сравниваются (это локальное упорядочивание).

### 6.8. Поведение АШ при работе с файлом

АШ работает **только со своим участком кода**. Остальной код в файле не должен изменяться.

Алгоритм работы модели:

1. Если в АШ указаны зависимости от других АШ (`depends_on`) — модель видит файл в состоянии после применения этих зависимостей.
2. Если зависимости не указаны — модель ищет в файле свой код по сигнатуре/имени из промпта.
3. Если свой код найден — модель его модифицирует.
4. Если не найден — модель дописывает свой код в конец файла.

### 6.9. Стоп-критерий разбиения

АШ дробятся до тех пор, пока **полный контекст одного шага** не помещается в окно модели-кодера.

Полный контекст = необходимая выжимка машинночитаемого ТЗ + родительский ГШ + родительский ТШ + текущее содержимое целевого файла + все необходимые сигнатуры, типы, импорты + сам промпт.

Целевой бюджет — 4к токенов. Это не привязка к конкретной модели, а **порог детализации**: показатель, насколько мелко надо дробить, чтобы маленькая локальная модель справилась без догадок.

Если контекст не помещается:

- Сначала пересматривается АШ — нельзя ли его раздробить.
- Если дробить уже некуда (один логический шаг физически не влезает) — это сигнал, что ТШ слишком крупный или файл слишком большой и должен быть разбит.

### 6.10. Самодостаточность промпта

Промпт АШ должен содержать всё, что нужно для выполнения, и ничего сверх. Модель не делает предположений, не читает другие файлы, не принимает проектных решений.

В промпт обязательно входят:

- Точный путь целевого файла.
- Тип операции из закрытого списка (см. 6.6).
- Полное текущее содержимое файла, если это модификация (с учётом применённых зависимостей).
- Все импорты и типы, которые будут использоваться.
- Точная сигнатура функции/класса.
- Требования к docstring и типизации.
- Все ограничения, явно перечисленные.
- Точечная выжимка концепций машинночитаемого ТЗ, нужных для шага. Не вся выжимка целиком, но и не ссылки — модель не должна никуда ходить.

В промпт **не** входят:

- Ссылки на другие шаги.
- Инструкции «прочитай такой-то файл».
- Открытые проектные решения.
- Формулировки вида «реализуй подходящим образом», «добавь нужные импорты».

### 6.11. Размер артефакта АШ

Файл АШ содержит только то, что необходимо для выполнения шага. Никакой воды, никакого дублирования между разными АШ. Размер минимизируется при сохранении полноты контекста.

### 6.12. Содержимое АШ

Обязательные поля:

- `step_id` — идентификатор формата `A-NNN`.
- `parent_tactical_step` — идентификатор родительского ТШ.
- `name` — краткое имя.
- `target_file` — путь целевого файла кода.
- `operation` — одно из: `create_file`, `modify_file`, `delete_file`, `rename_file`.
- `priority` — целое число (для упорядочивания при нескольких АШ на один файл).
- `depends_on` — опциональный список А-шагов, после которых выполняется этот.
- `concepts` — список идентификаторов концепций машинночитаемого ТЗ, упомянутых в промпте.
- `prompt` — полный промпт для модели-кодера (см. 6.10).
- `verification` — структурированное описание проверки (см. 6.13).
- `status` — текущий статус (см. раздел 9).

### 6.13. Верификация АШ

Структурированное описание проверки. Без кода — код пишется в других АШ (например, в АШ, создающем тестовый файл).

Поля:

- `type` — тип проверки из закрытого списка: `pytest`, `import`, `static_analysis`, `manual`.
- `target` — что именно проверяется (путь файла, имя теста, имя сущности).
- `expected` — ожидаемый результат.

Точный формат для каждого типа фиксируется при переводе стандарта в машинночитаемую форму.

---

## 7. Сквозные инварианты

Три свойства, которые должны выполняться всегда. На них держится корректность всей конструкции.

### 7.1. I1 — Покрытие

> Сумма потомков покрывает родителя без пропусков.

Применение по уровням:

- **a)** Объединение концепций всех ГШ ≡ множество концепций машинночитаемого ТЗ.
- **b)** Объединение связей, реализуемых в ГШ, ≡ множество связей машинночитаемого ТЗ.
- **c)** Объединение диапазонов строк всех ГШ ≡ значимые строки исходного ТЗ.

Первые два — автоматизируемы (операции над множествами). Третье — резервная проверка на случай неполноты машинночитаемого ТЗ.

**Когда проверяется:** перед заморозкой набора ГШ.

### 7.2. I2 — Семантическое покрытие ветки

> Набор АШ, выведенный из ветки машинночитаемое ТЗ + ГШ + ТШ, по смыслу полностью воспроизводит эту ветку.

Единственность набора АШ **не требуется**. Допустимо, что разные декомпозиции дают разные наборы АШ при условии, что каждый из них семантически покрывает ветку.

**Процедура проверки I2 (чек-лист).** Для каждого ТШ:

1. **Объекты.** Каждый объект из родительской ветки (концепции из машинночитаемого ТЗ + объекты, упомянутые в ГШ и ТШ) присутствует хотя бы в одном АШ — либо как создаваемый, либо как изменяемый.
2. **Операции.** Каждая операция из ветки реализуется хотя бы одним АШ.
3. **Связи.** Каждая связь между концепциями, относящаяся к этому ТШ, реализуется хотя бы одним АШ.
4. **Полнота промптов.** В каждом АШ перечислены все нужные ему концепции, типы, сигнатуры. Модель не должна предполагать.
5. **Отсутствие лишнего.** Ни один АШ не реализует функциональность вне ветки.

Если чек-лист пройден — I2 выполнен. Альтернативные декомпозиции этого же ТШ допустимы, если они также проходят чек-лист.

**Когда проверяется:** перед заморозкой набора АШ для каждого ТШ.

### 7.3. I3 — Эквивалентность

> Выполнение всех АШ реализует исходное ТЗ полностью. Никакая реализационная деталь не существует вне АШ.

Концепция или связь, не реализованная ни одним АШ, — это нарушение I3.

**Когда проверяется:** при завершении реализации (тесты + ручная проверка).

## 8. Каскадная процедура правки

### 8.1. Принцип

Любая правка артефактов плана идёт строго сверху вниз:

1. Исходное ТЗ.
2. Машинночитаемое ТЗ.
3. Глобальные шаги.
4. Тактические шаги (затронутые).
5. Атомарные шаги (затронутые).

Прямая правка нижестоящих артефактов в обход вышестоящих **запрещена без исключений**. Никаких `corrigendum`, никаких косметических правок мимо каскада.

### 8.2. Транзакционность

Каскад — атомарная транзакция. Пока не пройдены все шаги и не проверены инварианты, новое состояние не публикуется. Промежуточные состояния не видны нижним уровням и потребителям плана.

Технически это значит: работа идёт в отдельной ветке/копии, мерж в основное состояние — только при полном прохождении каскада.

### 8.3. Гранулярность инвалидации

Единое правило:

> При изменении любого узла дерева плана **все его прямые потомки** переходят в `needs_review`, должны быть проверены и при необходимости изменены. Изменения распространяются вниз каскадно: проверенные потомки в свою очередь могут оказаться изменёнными, что инвалидирует их собственных потомков.

Применение по уровням:

- Изменилось исходное ТЗ → машинночитаемое ТЗ перепроецируется (это его свойство как проекции).
- Изменилось машинночитаемое ТЗ → все ГШ переходят в `needs_review`.
- Изменился конкретный ГШ → все его ТШ переходят в `needs_review`. Соседние ГШ не затрагиваются.
- Изменился конкретный ТШ → все его АШ переходят в `needs_review`. Соседние ТШ не затрагиваются.

Поскольку прямая правка машинночитаемого ТЗ запрещена, все изменения концепций и связей идут от исходного ТЗ. Поэтому отдельных правил «инвалидация через ссылки на концепции» не нужно — они покрываются общим правилом.

### 8.4. Процедура каскада (пошагово)

**Шаг 1. Локализация в исходном ТЗ.** Изменение вносится в значимую часть исходного ТЗ.

**Шаг 2. Перепроекция в машинночитаемое ТЗ.** Машинночитаемое ТЗ обновляется. Получается diff на уровне концепций и связей: какие добавлены, какие удалены, какие изменили свойства, какие изменили связи.

**Шаг 3. Полная переоценка ГШ.** Все ГШ перепроверяются против обновлённого машинночитаемого ТЗ:

- Концепции, появившиеся после правки, распределяются по ГШ (возможно, в новый ГШ).
- Концепции, исчезнувшие после правки, удаляются из ГШ.
- Если границы концепций сместились — границы ГШ смещаются вместе с ними.
- Проверяется I1: новое объединение концепций ГШ снова покрывает машинночитаемое ТЗ.

Каждый ГШ помечается как изменённый или не изменённый по результату переоценки.

**Шаг 4. Выборочная ревизия ТШ.** Для каждого изменённого ГШ — все его ТШ переходят в `needs_review`. Ревизия ТШ может в свою очередь изменить их.

**Шаг 5. Выборочная ревизия АШ.** Для каждого изменённого ТШ — все его АШ переходят в `needs_review`. Ревизия может изменить промпты, добавить или удалить АШ.

**Шаг 6. Проверка инвариантов на новой версии.**

- I1 — после шага 3.
- I2 — после шага 5.
- I3 — после реализации правок в коде.

**Шаг 7. Публикация.** Если все инварианты прошли — новое состояние публикуется (мерж транзакции). Если нет — каскад откатывается, состояние не меняется.

### 8.5. Технический контроль каскада

На текущем этапе соблюдение правила обеспечивается **дисциплиной автора плана**. Контроль постфактум возможен через git-историю.

В перспективе вводится **Менеджер планирования** — отдельный инструмент, обеспечивающий технический контроль каскада и автоматизацию проверки инвариантов. До его появления стандарт опирается только на правило.

## 9. Статусы артефактов

Применимы ко всем уровням:

- `draft` — артефакт создаётся.
- `ready_for_review` — готов к проверке инварианта своего уровня.
- `frozen` — прошёл проверку, на него можно опираться нижестоящим уровням.
- `needs_review` — родительский уровень изменился, требует переоценки.

Дополнительно для АШ:

- `in_progress` — АШ в реализации.
- `done` — АШ реализован, верификация пройдена.

Статус `obsolete` не используется — старые артефакты при каскаде перезаписываются, история ведётся в git.

## 10. Идентификаторы и именование

### 10.1. Форматы

- ГШ: `G-NNN`, где `NNN` — три цифры с ведущими нулями (`G-001`, `G-012`).
- ТШ: `T-NNN`, формат аналогичный.
- АШ: `A-NNN`, формат аналогичный.
- Концепции машинночитаемого ТЗ: формат фиксируется при переводе в машинночитаемую форму.

### 10.2. Уникальность

- `G-NNN` уникален в пределах плана.
- `T-NNN` уникален в пределах своего родительского ГШ.
- `A-NNN` уникален в пределах своего родительского ТШ.

### 10.3. Имена директорий

```
G-NNN-<slug>/
T-NNN-<slug>/
```

`<slug>` — kebab-case, краткое имя шага.

### 10.4. Имена файлов

`README.yaml` для ГШ и ТШ.

## 12. Что описывает стандарт и что не описывает

**Стандарт описывает:**

- Структуру артефактов плана.
- Правила декомпозиции.
- Инварианты согласованности.
- Каскадную процедуру изменений.
- Статусы и идентификаторы.

**Стандарт не описывает:**

- Процедуру исполнения АШ моделью-кодером (когда, как, в каком окружении). Это уйдёт в отдельный документ про Менеджер планирования.
- Правила «одна модель работает с одним файлом одновременно» — это правило исполнения, не свойство плана. В стандарте плана остаётся только следствие: «АШ внутри одного файла кода имеют различающиеся приоритеты».
- Конкретные форматы YAML для машинночитаемой формы. Они фиксируются на следующем шаге перевода стандарта.
- Инфраструктуру разработки (CI, тесты, окружение) — это предмет других документов.
